
## Running All Unit Tests

The full test suite runs 294 unit tests via the project run.sh script:

```bash
cd /home/nick/work/libnn/project/ax45mpv
export QEMU_SYS_RV64=/local/nick/qemu_andes/build/qemu-system-riscv64
export PATH="/local/nick/SW_Release_cp/ast530/nds64le-elf-newlib-v5d/bin:/home/nick/work/libnn:$PATH"
```

## Running a Single Test

To quickly validate a single function after editing:

```bash
export QEMU_SYS_RV64=/local/nick/qemu_andes/build/qemu-system-riscv64
export PATH="/local/nick/SW_Release_cp/ast530/nds64le-elf-newlib-v5d/bin:/home/nick/work/libnn:$PATH"

# 1. Rebuild the library
cd /home/nick/work/libnn
build.sh AX45MPV

# 2. Compile the test
cd /home/nick/work/libnn/project/ax45mpv
CC=riscv64-elf-gcc
CFLAGS="-mtune=andes-45-series -mext-vector -DNDS_VEC_RVV_VERSION=1000 -O3 -Wall -Werror -ffunction-sections -fdata-sections -DPF_COUNTER -Wl,-gc-sections -DENA_FS_VS -mvh -DFAST_VERSION -DFUNCTION_PF_ANALYZE -mzfh -DENA_VL4"

$CC $CFLAGS $INC -o t_${FUNC_NAME}.adx ../../Examples/unit_func/t_${FUNC_NAME}.c $LIB

# 3. Run via QEMU
./riscv-qemu-wrapper-v1024_d1024_b512 t_${FUNC_NAME}.adx
```

Expected output contains `pass` for successful accuracy verification.

## Workflow: Validate After Optimization

Typical workflow after modifying a function's RVV implementation:

1. **Rebuild library**: `cd /home/nick/work/libnn && build.sh AX45MPV`
2. **Quick single-function test**: Compile and run the specific test via QEMU (see above)
3. **Full regression**: Run all 294 tests via `run.sh` to check nothing else broke
4. **Performance measurement**: Use the `fpga-bench` skill on FPGA board for cycle counts

- `pass` — function output matches golden reference data
- `fail` — output differs from golden data (accuracy regression)
- No `pass`/`fail` — test binary failed to execute (build or QEMU issue)
- Test golden data is in `Examples/unit_func/bin/`
- Test source files follow `Examples/unit_func/t_<function_name>.c` naming

# libnn RVV Optimization Loop

Systematic optimization workflow: **Idea → Implement → Verify (QEMU) → Benchmark (FPGA) → Analyze → Learn**

Every iteration — whether it improves or regresses — MUST end with a lesson stored in the optimization database. Bad results are as valuable as good ones.

## Phase 1: Idea

### Find optimization targets

```bash
# Search past lessons for relevant experience
.venv/bin/python3 scripts/optdb.py search "convolution small channels"

# Check full-suite performance data for hot functions
python3 scripts/compare_perf.py old/performance.csv new/performance.csv --top 20

# Review assembly of candidate function
grep -n "<function_name>" libnn.objdump
```

### Common optimization patterns (from experience DB)

Before proposing a new approach, search the vector DB:
```bash
.venv/bin/python3 scripts/optdb.py search "<your optimization idea>"
```

Key patterns discovered so far:
- **Vectorization axis**: When out_ch is small relative to VLEN, vectorize along reduction dimension instead of output channels
- **Existing kernels**: Search codebase for existing optimized kernels before writing new ones (e.g., nn_mat_mult_kernel_s8_offset_v2 already had vd4dots)
- **Dispatch thresholds**: Add runtime dispatch (if/else) to select best kernel based on tensor dimensions

### Validate the idea

Before implementing, answer:
1. What is the current bottleneck? (compute-bound vs memory-bound)
2. What % of vector capacity is utilized? (VLEN × LMUL vs actual elements)
3. Does an existing kernel already solve this pattern?
4. What speedup do you expect, and why?

## Phase 2: Implement

### Edit source code

Source files are under `Source/<Category>/`. Key conventions:
- Vector code is guarded by `#ifdef ENA_VEC_ISA` and `NDS_VEC_RVV_VERSION`
- Use macros from `internal/internal_vec_i_v1_0.h` for ACE instructions (e.g., `NDS_VEC_VD4DOTS_VV`)
- Target VLEN=1024, but code must handle any VLEN via strip-mining

### Build the library

```bash
cd /home/nick/work/libnn/qemu-validate
bash build_libnn.sh --verbose
```

Build must succeed with zero warnings (`-Wall -Werror`).

## Phase 3: Verify (QEMU)

Functional correctness check — MUST pass before FPGA benchmarking.

```bash
cd /home/nick/work/libnn/qemu-validate
bash run_qemu_test.sh <function_name> --strace
```

**IMPORTANT**: Always use `--strace` when running from Claude Code (QEMU semihosting output goes to a non-stdout fd in sandboxed environments).

Expected output: `accuracy checking pass` for all sub-tests (typically 3).

If FAIL: fix the code and re-verify. Do NOT proceed to FPGA with failing tests.

## Phase 4: Benchmark (FPGA)

Cycle-accurate measurement on real hardware.

### A/B comparison

```bash
cd /home/nick/work/libnn/qemu-validate

# Baseline (before optimization — skip if you already have baseline numbers)
# git stash; bash build_libnn.sh --verbose
# bash run_fpga_test.sh <function_name> --perf --tag baseline
# git stash pop

# Optimized
bash build_libnn.sh --verbose
bash run_fpga_test.sh <function_name> --perf --tag optimized

# Compare
cat work/<function_name>_baseline.log
cat work/<function_name>_optimized.log
```

### Reading results

```
test:   <function_name>
tag:    optimized
inst:   1039912          ← instruction count
cycles: 2187394          ← cycle count (THIS is the ground truth)
result: PASS             ← must be PASS
```

Board variance is typically <1%. If >5%, suspect thermal or contention (try different port).

## Phase 5: Analyze — THE CRITICAL STEP

Whether the result is good or bad, you MUST analyze WHY.

### If improved: understand the real cause

Don't assume — verify:
1. **Review objdump**: Did the compiler emit what you expected?
   ```bash
   grep -n "<function_name>" libnn.objdump
   # Check hot loop: correct LMUL? expected instructions? good scheduling?
   ```

2. **Check instruction reduction**: Fewer instructions often matters more than better compute utilization
   ```
   Baseline:  inst=3826922  cycles=5456391
   Optimized: inst=1039912  cycles=2187394
   → 3.68x fewer instructions, 2.49x fewer cycles
   ```

3. **vsim pipeline profiling** (optional, for deeper understanding):
   ```bash
   cd qemu-validate/vsim-bench
   bash run_vsim_pipeline.sh test_program.c --pipeline
   # Note: vsim is core-only, no memory model. Use for relative pipeline insight only.
   ```

### If regressed or no improvement: understand why the idea failed

This is MORE valuable than a success — it prevents repeating the same mistake.

Common failure causes:
- Compiler didn't emit expected code (check objdump)
- Memory-bound: better compute doesn't help if loads dominate
- Register pressure: high LMUL causes spills that negate compute gains
- Branch misprediction from added dispatch logic
- im2col overhead dominates (copying data costs more than computing)

## Phase 6: Learn — Store in Vector Database

**MANDATORY**: Every optimization attempt gets a database entry.

```bash
cd /home/nick/work/libnn

# Success example
.venv/bin/python3 scripts/optdb.py add \
    --title "Vectorize conv GEMM along reduction axis for small out_ch" \
    --category vectorization \
    --tags "vd4dots,gemm,conv,axis-selection" \
    --function "riscv_nn_conv_HWC_s8_s8_s8_asym_bias_any" \
    --speedup "2.49x" \
    --fpga-cycles "5456391 → 2187394" \
    --text "When out_tensor_ch=16 with VLEN=1024, vectorizing across channels wastes 87.5% of vector capacity. Switching to vectorize along col (reduction) dimension with vd4dots gave 2.49x speedup. Key: reused existing v2 kernel, just added dispatch."

# Failure example
.venv/bin/python3 scripts/optdb.py add \
    --title "LMUL=8 vd4dots attempt — no improvement due to register pressure" \
    --category vectorization \
    --tags "lmul,register-pressure,failure" \
    --function "nn_mat_mult_kernel_s8_offset_v2" \
    --speedup "1.0x (no change)" \
    --text "Tried LMUL=8 to process more elements per vd4dots. Compiler spilled registers due to needing 8 register groups × 4 accumulators = 32 registers (entire VRF). Spill/reload overhead negated compute gains. Lesson: LMUL=4 is the sweet spot for 2×2 tiling — uses all 32 registers without spills."
```

# libnn RVV Optimization Skill

Fully autonomous optimization loop. State persisted in `.claude/opt-state.json`.

## Current State

!`cat /home/nick/work/libnn/.claude/opt-state.json 2>/dev/null || echo '{"step":"idle","function":"","round":0,"history":[]}'`

## Commands

- `/optimize <function_name>` — Start optimizing a function (loops until plateau)
- `/optimize status` — Show current state
- `/optimize reset` — Reset to idle

Arguments received: `$ARGUMENTS`

## Finite State Machine

```
idle → select → analyze → implement → verify ──→ measure → profile → learn ─┐
                  ↑          ↑          |                                     |
                  |          └──────────┘ (verify fail → fix → retry)        |
                  |                                                          |
                  └──────────────────────────────────────────────────────────┘
                    (loop with new pipeline insights + assembly analysis)

Exit condition: 2 consecutive rounds with no improvement → idle
```

The loop is the core mechanic: after `learn`, always go back to `analyze` with fresh insights from the pipeline profile and kernel assembly. Each round builds on what was learned.

---

## State: `idle`

No optimization in progress.

If a function name is provided, transition → `select`.
If "status", report idle. If "reset", clear state.

---

## State: `select`

Validate the function and gather baseline context.

1. Find the source file under `Source/` — grep for the function name
2. Confirm it has `#ifdef ENA_VEC_ISA` (existing RVV to improve)
3. Find corresponding test in `Examples/unit_func/t_<name>.c`
4. Query optdb for prior lessons:
   ```bash
   .venv/bin/python3 scripts/optdb.py search "<function_name>"
   ```
5. Build libnn and run a baseline FPGA measurement to get the "before" cycle count
6. Save state: function, source_file, test_name, cycles_before (baseline)

**Transition:** → `analyze`
**Fail:** Function not found or no RVV → stay `select`, report error.

---

## State: `analyze`

Study the current RVV code, assembly, and pipeline data to form the next optimization hypothesis.

**On first round (round=1):**
1. Read the full source file (RVV path and plain C fallback)
2. Build libnn, then find the function in the objdump:
   ```bash
   grep -A 200 '<function_name>:' libnn.objdump
   ```
3. Query optdb for lessons on similar patterns
4. Identify bottleneck: LMUL choice, tiling, register pressure, memory access, scheduling
5. Form hypothesis: what to change and expected impact

**On subsequent rounds (round>1):**
1. Re-read the source (it changed last round)
2. Re-check the objdump for the updated assembly
3. Review the pipeline profile from last round's `profile` step (stored in state as `pipeline_insight`)
4. Look for NEW bottlenecks revealed by the profile:
   - VQ→VD stalls → interleave scalar/vector work
   - Too many VW stages → reduce LMUL, increase tiling
   - Poor dual-issue → reorder instructions
   - VLSU stalls → add prefetch, align data
   - Register spills in assembly → reduce register pressure
5. Query optdb with the specific bottleneck pattern
6. Form the next hypothesis based on what the profile revealed

**Output:** Print the analysis summary and hypothesis. Do NOT wait for approval — proceed immediately.

**Transition:** → `implement`

## Finite State Machine

```
idle → select → analyze ──→ implement → verify ──→ measure → profile → learn ─┐
                  ↑ |          ↑          |                                     |
                  | |          └──────────┘ (verify fail → fix → retry)        |
                  | |                                                          |
                  | └→ idle (no opportunity found)                             |

```

The loop is the core mechanic: after `learn`, go back to `analyze` with fresh insights from the pipeline profile and kernel assembly. Each round builds on what was learned. # Wiki Writer

Internal skill: called by `/optimize` during the `learn` state. Defines style rules, page templates, and update operations for `docs/wiki/`.

## Purpose

Wiki pages serve two audiences: humans reading them for context, and the `/optimize` agent reading them as structured memory. Pages must be clear to both. The failure mode to avoid is jargon-only writing -- pages that assume the reader already knows what GEMM, LMUL, or vd4dots means.

**Three core principles:**

1. **Define every acronym on first use.** Write "GEMM (General Matrix Multiply)" the first time, then "GEMM" after.
2. **Lead with plain-English summary.** The first content section of every page must explain what this is in terms any engineer understands, with no assumed domain knowledge.
3. **Use concrete numbers, not vague words.** "2.1x speedup (4,823 to 2,298 cycles)" not "significant improvement". "IPC of 1.04 when the theoretical max is 2.0" not "low IPC".

---

## Jargon Glossary

Reference table for constructing inline definitions. Do not paste these verbatim -- incorporate them naturally into sentences.

| Term | Plain-English Expansion |
|------|------------------------|
| GEMM | General Matrix Multiply -- multiplying two matrices together; the core math inside convolution |
| im2col | "image to column" -- a transform that reshapes a 3D image patch into a flat row so convolution becomes a matrix multiply |
| LMUL | "length multiplier" -- how many physical vector registers are ganged together into one logical register; higher LMUL = more elements per instruction but fewer usable register names |
| VL | "vector length" -- how many elements the CPU actually processes in one instruction; set by vsetvli |
| VLEN | hardware vector register width in bits (e.g., 1024 bits = 128 bytes per register) |
| vsetvli | "vector set vector length immediate" -- instruction that sets VL and SEW for the next block of vector code |
| vd4dots | Andes custom dot-product instruction: computes 4 partial dot products per cycle using 4-element int8 groups |
| SEW | "selected element width" -- the bit width of one element in a vector register (8, 16, 32, or 64 bits) |
| VLSU | "vector load/store unit" -- the hardware pipeline stage that handles vector memory operations |
| IPC | "instructions per cycle" -- how many instructions finish per clock; theoretical max on AX45MPV is 2.0 for most code |
| VQ | vector quarter-width -- element width is VLEN/4 |
| VD | vector double-width -- element width is VLEN*2 |
| VC | vector compressed -- used in vector compress/expand operations |
| VW | "vector widening" -- an instruction that produces a result twice as wide as its inputs (e.g., 8-bit inputs, 16-bit output) |
| dual-issue | AX45MPV can retire two instructions per cycle when they have no data dependency and use different execution units |
| VREDSUM | "vector reduction sum" -- collapses all elements in a vector register into a single scalar by adding them |
| tiling | splitting a large computation into smaller blocks that fit in registers; trades loop overhead for better register reuse |

## Style Rules with Before/After Examples

### Rule 1: Explain technique impact with an analogy, not just a metric

**Before (jargon):**
> Switched from LMUL=1 to LMUL=4, increasing register utilization from 25% to 100%.

**After (naive expression):**
> The vector unit is like a bus with 128 seats. With LMUL=1 at VLEN=1024 processing int8, each instruction uses all 128 seats. But the old code used LMUL=4 with int32, which means only 32 elements per register -- like running that bus with 32 passengers. We switched to packing 4x more work per instruction.

The analogy makes the performance lever visible. Lead with the analogy, follow with the numbers.

### Rule 2: Name anti-pattern symptoms in plain terms

**Before (jargon):**
> vd4dots exhibits poor utilization when K < 16 due to partial group completion.

**After (naive expression):**
> vd4dots works by grouping 4 input channels at a time. When the number of input channels (K) is less than 16, most groups are incomplete -- like a conveyor belt that moves 4 items at a time but only has 1-3 items on most trips. The hardware still pays the full cost per group, so efficiency drops sharply below K=16.

### Rule 3: Explain what pipeline numbers mean, not just what they are

**Before (jargon):**
> Profile showed IPC=1.04 with high VLSU stalls, indicating memory-bound execution.

**After (naive expression):**
> The profiler measured 1.04 instructions completed per clock cycle. The AX45MPV can theoretically do 2.0 per cycle, so we are using about half the available throughput. The stall log showed the vector load/store unit (VLSU) was blocked waiting for data from memory on nearly every other cycle -- the computation was fast but data was arriving too slowly to keep it fed.

### Rule 4: Code comments must explain purpose, not restate the instruction

**Before:**
```c
vsetvli(t0, a0, e8, m4);   // vsetvli
```

**After:**
```c
// Set VL to process up to 4*VLEN/8 int8 elements per iteration.
```

## State: `analyze`

Study the current RVV code, assembly, and pipeline data to form the next optimization hypothesis.

**On first round (round=1):**
1. Read the full source file (RVV path and plain C fallback)
2. Find the function in `libnn.objdump`:

3. Query optdb for lessons on similar patterns

2. Re-check the objdump for the updated assembly

Before any optimization work, validate that config.toml and all referenced toolchain/QEMU/FPGA resources are accessible.

Run:
```bash
python3 /home/nick/work/agentic_libnn_opt/opt/script/validate_config.py
```

This runs 7 sequential checks, failing fast on the first error:
1. **Load config.toml** - all required sections and keys exist
2. **Toolchain** - bin_dir exists, gcc compiles with Andes flags
3. **Compile hello world** - cross-compilation produces RISC-V ELF

## Phase 1: Gather

Parse the task specification and target features. No builds.

1. Read `opt/config.toml`. Extract toolchain PATH, QEMU binary, FPGA host:port, CPU, VLEN.

2. Read the FPGA `.inc` file (path from `[target].fpga_config`, relative to `opt/`). Extract:

3. Read `state.source_input`:

   - **Model mode:** parse model `.c` and included param headers. Build ordered `(function, shape-set)` list.

Measure unmodified library on the target.

2. QEMU correctness: `/usr/bin/bash /home/nick/work/libnn/qemu-validate/run_qemu_test.sh <test> --strace --vlen <vlen>`

## Phase 3: Iterate

One iteration = one candidate optimization attempt.

1. **Plan** - Inspect current best source (or baseline on iter 1). Pick one concrete change.
2. **Edit** - Modify source under `#ifdef ENA_AGENT_OPT_<FUNC>`. Baseline `#else` stays untouched.
3. **Build** - Rebuild with and without the macro flag.

7. **Accept/reject** - `accepted = pass && cycles < best.cycles`. Update `best` if accepted.
8. Append to `history`. Persist state.json.
