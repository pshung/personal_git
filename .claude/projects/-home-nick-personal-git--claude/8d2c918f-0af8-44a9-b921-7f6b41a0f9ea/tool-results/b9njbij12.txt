# Commit Optimizer Skill

Creates well-documented commit messages for optimization work with automatic benchmarking.

## Usage

```
/commit [problem="<description>"] [approach="<description>"]
```

## How It Works

1. **Validates optimization context** — checks that you're in an optimization commit (staged changes present)
2. **Detects what changed** — examines git diff to understand which functions/files were modified
3. **Runs benchmarks** — executes the project's measurement tools (Makefile, commit-size script, or QEMU tests) to capture before/after metrics
4. **Builds commit message** — combines problem statement, approach, and measured results into a structured message

## Commit Message Format

The skill generates messages in this format:

```
optimize: <one-line summary>

Problem:
<explanation of what was slow/large and why it mattered>

Approach:
<description of the optimization technique used>

Result:
<measured improvement with specific metrics>
```

## Examples

**Example 1: Assembly leaf function optimization**
```
optimize: sf_floor.s as leaf function (remove ra push/pop)

Problem:
sf_floor saved and restored return address (ra) unnecessarily, adding 2 instructions (~8 bytes) and extra memory traffic on every call.

Approach:
Eliminated ra push/pop by using tail-call pattern; function now only uses caller-saved registers and doesn't need to preserve ra.

Result:
Code size: 24 → 16 bytes (-33%)
No performance loss (leaf function overhead removed)
```

**Example 2: Register allocation tuning**
```
optimize: sinf.s improve register allocation for FPU operations

Problem:
Register spills were causing extra load/store operations in the hot path of sinf, adding latency on FPU-heavy workloads.

Approach:
Reordered argument reduction to keep intermediate values in registers longer; used callee-saved registers more strategically to reduce spill pressure.

Result:
Code size: 312 → 298 bytes (-4%)
Execution time: 142 → 136 cycles (-4%)
```

## What the Skill Does

- **Auto-detects function** from changed files (e.g., `sf_floor.s` → optimizing `sf_floor`)
- **Runs project benchmarks** to measure size/performance before/after
- **Extracts metrics** from benchmark output (code size, cycle count, etc.)
- **Allows customization** — you can provide problem and approach strings to override auto-detection

## When NOT to Use

This skill is **only** for optimization commits. Use regular git commit for:
- Bug fixes (unless they're performance-related)
- Refactoring without size/performance gains
- Documentation changes
- Adding new features

## Required Parameters (optional)

- `problem` — Brief explanation of what was suboptimal (auto-detected from function changes if omitted)
- `approach` — Description of the optimization technique (auto-detected from git diff if omitted)

The benchmark result is **always** captured automatically.

## Integration with Your Project

This skill is designed for Picolibc RISC-V libm work:
- Looks for Makefile `compare` target to run benchmarks
- Falls back to `commit-size` script if available
- Parses QEMU output or size tool output to extract metrics
- Works with both `-s` assembly files and C implementations in `hardfloat/`

# Wiki Writer

Internal skill: called by `/optimize` during the `learn` state. Defines style rules, page templates, and update operations for `docs/wiki/`.

## Purpose

Wiki pages serve two audiences: humans reading them for context, and the `/optimize` agent reading them as structured memory. Pages must be clear to both. The failure mode to avoid is jargon-only writing -- pages that assume the reader already knows what GEMM, LMUL, or vd4dots means.

**Three core principles:**

1. **Define every acronym on first use.** Write "GEMM (General Matrix Multiply)" the first time, then "GEMM" after.
2. **Lead with plain-English summary.** The first content section of every page must explain what this is in terms any engineer understands, with no assumed domain knowledge.
3. **Use concrete numbers, not vague words.** "2.1x speedup (4,823 to 2,298 cycles)" not "significant improvement". "IPC of 1.04 when the theoretical max is 2.0" not "low IPC".

---

## Jargon Glossary

Reference table for constructing inline definitions. Do not paste these verbatim -- incorporate them naturally into sentences.

| Term | Plain-English Expansion |
|------|------------------------|
| GEMM | General Matrix Multiply -- multiplying two matrices together; the core math inside convolution |
| im2col | "image to column" -- a transform that reshapes a 3D image patch into a flat row so convolution becomes a matrix multiply |
| LMUL | "length multiplier" -- how many physical vector registers are ganged together into one logical register; higher LMUL = more elements per instruction but fewer usable register names |
| VL | "vector length" -- how many elements the CPU actually processes in one instruction; set by vsetvli |
| VLEN | hardware vector register width in bits (e.g., 1024 bits = 128 bytes per register) |
| vsetvli | "vector set vector length immediate" -- instruction that sets VL and SEW for the next block of vector code |
| vd4dots | Andes custom dot-product instruction: computes 4 partial dot products per cycle using 4-element int8 groups |
| SEW | "selected element width" -- the bit width of one element in a vector register (8, 16, 32, or 64 bits) |
| VLSU | "vector load/store unit" -- the hardware pipeline stage that handles vector memory operations |
| IPC | "instructions per cycle" -- how many instructions finish per clock; theoretical max on AX45MPV is 2.0 for most code |
| VQ | vector quarter-width -- element width is VLEN/4 |
| VD | vector double-width -- element width is VLEN*2 |
| VC | vector compressed -- used in vector compress/expand operations |
| VW | "vector widening" -- an instruction that produces a result twice as wide as its inputs (e.g., 8-bit inputs, 16-bit output) |
| dual-issue | AX45MPV can retire two instructions per cycle when they have no data dependency and use different execution units |
| VREDSUM | "vector reduction sum" -- collapses all elements in a vector register into a single scalar by adding them |
| tiling | splitting a large computation into smaller blocks that fit in registers; trades loop overhead for better register reuse |

## Style Rules with Before/After Examples

### Rule 1: Explain technique impact with an analogy, not just a metric

**Before (jargon):**
> Switched from LMUL=1 to LMUL=4, increasing register utilization from 25% to 100%.

**After (naive expression):**
> The vector unit is like a bus with 128 seats. With LMUL=1 at VLEN=1024 processing int8, each instruction uses all 128 seats. But the old code used LMUL=4 with int32, which means only 32 elements per register -- like running that bus with 32 passengers. We switched to packing 4x more work per instruction.

The analogy makes the performance lever visible. Lead with the analogy, follow with the numbers.

### Rule 2: Name anti-pattern symptoms in plain terms

**Before (jargon):**
> vd4dots exhibits poor utilization when K < 16 due to partial group completion.

**After (naive expression):**
> vd4dots works by grouping 4 input channels at a time. When the number of input channels (K) is less than 16, most groups are incomplete -- like a conveyor belt that moves 4 items at a time but only has 1-3 items on most trips. The hardware still pays the full cost per group, so efficiency drops sharply below K=16.

### Rule 3: Explain what pipeline numbers mean, not just what they are

**Before (jargon):**
> Profile showed IPC=1.04 with high VLSU stalls, indicating memory-bound execution.

**After (naive expression):**
> The profiler measured 1.04 instructions completed per clock cycle. The AX45MPV can theoretically do 2.0 per cycle, so we are using about half the available throughput. The stall log showed the vector load/store unit (VLSU) was blocked waiting for data from memory on nearly every other cycle -- the computation was fast but data was arriving too slowly to keep it fed.

### Rule 4: Code comments must explain purpose, not restate the instruction

**Before:**
```c
vsetvli(t0, a0, e8, m4);   // vsetvli
```

**After:**
```c
// Set VL to process up to 4*VLEN/8 int8 elements per iteration.
// LMUL=4 groups 4 registers together -- trades register name space
// for throughput when K is large enough to fill all 4 registers.
vsetvli(t0, a0, e8, m4);
```

# libnn Optimizer

Drives autonomous optimization of libnn NN operator functions targeting RISC-V FPGA hardware.
The workflow has 5 phases: gather, baseline, iterate, evaluate, done.

## Project Paths

- **Optimization root:** `/home/nick/work/agentic_libnn_opt/opt/`
- **Config:** `opt/config.toml` (user-owned, NEVER modify)
- **State:** `opt/state.json` (LLM reads/writes to track progress)
- **Scripts:** `opt/script/`
- **Plan:** `/home/nick/work/agentic_libnn_opt/plan.md`
- **libnn source:** `/home/nick/work/libnn/`
- **FPGA configs:** `opt/fpga_config/`

## Step 0: Validate Configuration

Before any optimization work, validate that config.toml and all referenced toolchain/QEMU/FPGA resources are accessible.

Run:
```bash
python3 /home/nick/work/agentic_libnn_opt/opt/script/validate_config.py
```

This runs 7 sequential checks, failing fast on the first error:
1. **Load config.toml** - all required sections and keys exist
2. **Toolchain** - bin_dir exists, gcc compiles with Andes flags
3. **Compile hello world** - cross-compilation produces RISC-V ELF
4. **QEMU** - binary exists, runs hello world via semihosting (strace mode)
5. **FPGA .inc config** - parses hardware features (VLEN, cache sizes, ISA extensions)
6. **FPGA connectivity** - TCP connection to board host:port
7. **FPGA run** - compiles and runs hello world on FPGA board

If validation fails at Step 2 (toolchain), the script suggests alternative paths found on the system. Report the suggestion to the user but do NOT modify config.toml -- it is user-owned.

If QEMU or FPGA steps fail due to missing binaries or network issues, report clearly and ask the user which steps to skip.

## Phase 1: Gather

Parse the task specification and target features. No builds.

1. Read `opt/config.toml`. Extract toolchain PATH, QEMU binary, FPGA host:port, CPU, VLEN.

2. Read the FPGA `.inc` file (path from `[target].fpga_config`, relative to `opt/`). Extract:
   - Vector: `NDS_VLEN`, `NDS_DLEN`, `NDS_ELEN`
   - Cache: `NDS_DCACHE_SIZE_KB`, `NDS_DCACHE_WAY`, `NDS_ICACHE_SIZE_KB`, `NDS_CACHE_LINE_SIZE`
   - L2: `NDS_L2C_CACHE_SIZE_KB`, `NDS_L2C_WAY`, `NDS_L2_DATA_WIDTH`
   - ISA: `NDS_RVV_SUPPORT`, `NDS_VECTOR_VDOT_SUPPORT`, `NDS_INT4_VECTOR_LOAD_SUPPORT`, `NDS_BFLOAT16_SUPPORT`, `NDS_QMAC_SUPPORT`
   - Bus: `NDS_BIU_DATA_WIDTH`

3. Read `state.source_input`:
   - **Single mode:** parse `#define` block from perf test `.c` file. Extract shapes (INPUT_X, INPUT_Y, INPUT_CH, KERNEL_X, etc.), quantization params (OUT_SHIFT, OUT_MULT, offsets), and resolve target function name.
   - **Model mode:** parse model `.c` and included param headers. Build ordered `(function, shape-set)` list.

4. Locate baseline source at `/home/nick/work/libnn/Source/<Category>/riscv_nn_<func>.c`.

5. Write all extracted data to `opt/state.json`. Set `phase` to `"baseline"`.

**Verification:** For `t_pf_conv_dw_HWC_u8_u8_u8_asym_bias_any.c`, state.json must contain the exact shapes from the `#define` block and `target_features.vlen == 1024`, `target_features.dcache_kb == 32`, `target_features.l2_kb == 8192`.

## Phase 2: Baseline

Measure unmodified library on the target.

1. Rebuild baseline: `/usr/bin/bash /home/nick/work/libnn/build.sh <cpu>`
2. QEMU correctness: `/usr/bin/bash /home/nick/work/libnn/qemu-validate/run_qemu_test.sh <test> --strace --vlen <vlen>`
3. FPGA perf: `/usr/bin/bash /home/nick/work/libnn/test_perf.sh ax45mpv riscv64-elf-gcc BOARD <function>`
4. Write `baseline` cycles/instructions to state.json. Set `phase` to `"iterate"`.

## Phase 3: Iterate

One iteration = one candidate optimization attempt.

1. **Plan** - Inspect current best source (or baseline on iter 1). Pick one concrete change.
2. **Edit** - Modify source under `#ifdef ENA_AGENT_OPT_<FUNC>`. Baseline `#else` stays untouched.
3. **Build** - Rebuild with and without the macro flag.
4. **Correctness** - QEMU strace test. Fail -> `accepted=false`, skip perf.
5. **Perf** - FPGA `test_perf.sh`. Collect cycles/instructions.
6. **Commit** - Git commit in libnn tree. Snapshot diff to `opt/patches/`.
7. **Accept/reject** - `accepted = pass && cycles < best.cycles`. Update `best` if accepted.
8. Append to `history`. Persist state.json.

Loop until `max_iterations` or no improvement >= `min_improvement_pct` for 2 consecutive attempts.

## Phase 4: Evaluate

1. Restore `best.commit` in libnn worktree.
2. Rerun full QEMU suite for touched functions.
3. Rerun FPGA perf, confirm within +/- 0.5% of `best.cycles`.
4. Emit `opt/report.md` with per-pair baseline -> best cycles, speedup, rationale.
5. Set `phase` to `"done"`.

## Phase 5: Done

Terminal. Human reviews `opt/report.md`.

## State Machine

Read `opt/state.json` field `phase` to determine where to resume:
- `"gather"` -> run Phase 1
- `"baseline"` -> run Phase 2
- `"iterate"` -> run Phase 3
- `"evaluate"` -> run Phase 4
- `"done"` -> report complete

## Key Rules

- **Never modify `opt/config.toml`** -- it is user-owned configuration.
- **Baseline code is never modified** -- optimized paths go under `#ifdef ENA_AGENT_OPT_<FUNC>`.
- **QEMU is for correctness only** -- FPGA cycle counts are ground truth for performance.
- **Primary target: AX45MPV VLEN=1024** -- do not sacrifice VLEN=1024 perf for smaller VLENs.
- Macro naming: `ENA_AGENT_OPT_<FUNC>` (single mode) or `ENA_AGENT_OPT_<FUNC>_S<N>` (model mode).

## Phase 1: Gather

Parse task spec and target features. No builds.

1. **Read config.toml.** Extract toolchain PATH, QEMU binary, FPGA host:port, CPU, VLEN.

2. **Read FPGA .inc file** (path from `[target].fpga_config`, relative to `opt/`). Extract:
   - Vector: `NDS_VLEN`, `NDS_DLEN`, `NDS_ELEN`
   - D-cache: `NDS_DCACHE_SIZE_KB`, `NDS_DCACHE_WAY`
   - I-cache: `NDS_ICACHE_SIZE_KB`, `NDS_CACHE_LINE_SIZE`
   - L2: `NDS_L2C_CACHE_SIZE_KB`, `NDS_L2C_WAY`, `NDS_L2_DATA_WIDTH`
   - ISA: `NDS_RVV_SUPPORT`, `NDS_VECTOR_VDOT_SUPPORT`, `NDS_INT4_VECTOR_LOAD_SUPPORT`, `NDS_BFLOAT16_SUPPORT`, `NDS_QMAC_SUPPORT`
   - Bus: `NDS_BIU_DATA_WIDTH`

   Parse using regex: `` `define\s+<KEY>\s+(\d+) `` for integers, `` `define\s+<KEY>_INT\s+(\d+) `` or `` `define\s+<KEY>\s+"(yes|no)" `` for booleans.

3. **Read source_input** (perf test or model C file):
   - **Single mode:** parse `#define` block from `Examples/unit_performance/t_pf_<func>.c`. Extract shapes (INPUT_X, INPUT_Y, INPUT_CH, KERNEL_X, etc.), quantization params (OUT_SHIFT, OUT_MULT, offsets, activation min/max), buffer sizes. Resolve function name from `riscv_nn_*()` call site.
   - **Model mode:** parse model `.c` and `#include`d param headers. Build ordered `(function, shape-set)` list with call counts.

4. **Locate baseline source** at `/home/nick/work/libnn/Source/<Category>/riscv_nn_<func>.c`. Record path.

5. **Derive optimization macro:** `ENA_AGENT_OPT_<FUNC_UPPER>` (single) or `ENA_AGENT_OPT_<FUNC_UPPER>_S<N>` (model).

6. **Write state.json.** Set `phase` to `"baseline"`.

**Verification:** state.json must contain:
- All shapes from the perf test `#define` block
- `target_features.vlen == 1024`
- `target_features.dcache_kb == 32`
- `target_features.l2_kb == 8192`
- Correct `source_file` path under `Source/`

## Phase 2: Baseline

Measure unmodified library on target.

1. Rebuild baseline:
   ```bash
   /usr/bin/bash /home/nick/work/libnn/build.sh <cpu>
   ```

2. QEMU correctness (strace mode for semihosting fd 9):
   ```bash
   /usr/bin/bash /home/nick/work/libnn/qemu-validate/run_qemu_test.sh <test> --strace --vlen <vlen>
   ```
   Accept only "accuracy checking pass". Failure aborts -- baseline is broken.

3. FPGA perf measurement:
   ```bash
   mkdir -p /home/nick/work/libnn/build_perf && cd /home/nick/work/libnn/build_perf
   /usr/bin/bash /home/nick/work/libnn/test_perf.sh ax45mpv riscv64-elf-gcc BOARD <function>
   ```
   Parse `performance.csv` for cycles and instructions.

4. Write `baseline` to state.json. Initialize empty `history`. Set `phase` to `"iterate"`.

## Phase 3: Iterate

One iteration = one candidate optimization. Loop until `max_iterations` or no improvement >= `min_improvement_pct` for 2 consecutive attempts.

Per iteration:

1. **Plan** - Inspect current best source (or baseline on iter 1). Pick one concrete change. Record in `notes`.

2. **Edit** - Modify `Source/<Category>/riscv_nn_<func>.c`: add optimized path under `#ifdef ENA_AGENT_OPT_<FUNC>`. Replace prior iteration's block if present. Baseline `#else` stays byte-for-byte intact.

3. **Build** - Rebuild `libnn.a` twice:
   - Without macro (sanity -- baseline still compiles)
   - With `-DENA_AGENT_OPT_<FUNC>` for perf

4. **Correctness** - QEMU strace test. Fail -> `accepted=false`, skip perf.

5. **Perf** - `test_perf.sh`. Collect cycles/instructions.

6. **Commit** - Git commit in libnn tree. Snapshot diff to `opt/patches/iter_<N>_<func>.patch`.

7. **Accept/reject** - `accepted = pass && cycles < best.cycles`. Update `best` if accepted.

8. Append to `history`. Persist state.json.

## Phase 4: Evaluate

1. Restore `best.commit` in libnn worktree.
2. Rerun full QEMU suite for touched functions.
3. Rerun FPGA perf, confirm within +/- 0.5% of `best.cycles`.
4. Model mode only: re-profile all (function, shape-set) pairs.
5. Emit `opt/report.md`: per-pair baseline -> best cycles, speedup, commit SHA, rationale.
6. Set `phase` to `"done"`.

## Phase 5: Done

Terminal. Human reviews `opt/report.md`. Winning commits in libnn tree are the deliverable.

## State Machine

Read `opt/state.json` field `phase` to determine where to resume:

```
gather -> baseline -> iterate (loop) -> evaluate -> done
```

## Key Rules

1. **Never modify `opt/config.toml`** -- user-owned configuration.
2. **Baseline code is never modified** -- optimized paths go under `#ifdef ENA_AGENT_OPT_<FUNC>`.
3. **QEMU is for correctness only** -- FPGA cycle counts are ground truth.
4. **Primary target: AX45MPV VLEN=1024** -- do not sacrifice VLEN=1024 perf.
5. **Use `NDS_VEC_*` macros only** in `ENA_VEC_ISA` blocks. Never `__riscv_*` intrinsics.
6. Macro naming: `ENA_AGENT_OPT_<FUNC>` (single) or `ENA_AGENT_OPT_<FUNC>_S<N>` (model).
7. Always validate config before first build (Step 0).
8. Scripts are the single source of truth for builds and tests -- never construct gcc/QEMU/GDB commands manually.
