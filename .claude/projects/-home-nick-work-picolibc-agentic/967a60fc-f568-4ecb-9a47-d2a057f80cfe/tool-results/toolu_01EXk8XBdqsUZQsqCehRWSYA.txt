     1→# Picolibc
     2→Copyright © 2018-2023 Keith Packard
     3→
     4→Picolibc is library offering standard C library APIs that targets
     5→small embedded systems with limited RAM. Picolibc was formed by blending
     6→code from [Newlib](http://sourceware.org/newlib/) and
     7→[AVR Libc](https://www.nongnu.org/avr-libc/).

    36→## Supported Architectures
    37→
    38→Picolibc has integrated testing support for many architectures which
    39→is used to validate the code for all patch integration:
    40→
    41→ * ARC (32- and 64- bit)
    42→ * ARM (32- and 64- bit)
    43→ * i386 (Native and Linux hosted, for testing)
    44→ * LatticeMico32
    45→ * LoongArch
    46→ * Motorola 68000 (m68k)
    47→ * MIPS
    48→ * MSP430
    49→ * Nios II
    50→ * OpenRisc
    51→ * Power9
    52→ * Renesas RX
    53→ * RISC-V (both 32- and 64- bit)
    54→ * SparcV8 (32 bit)
    55→ * SuperH
    56→ * x86_64 (Native and Linux hosted, for testing)
    57→ * Xtensa ESP32
    58→
    59→There is also build infrastructure and continuous build validation,
    60→but no integrated testing available for additional architectures:
    61→
    62→ * Microblaze (32-bit, big and little endian)
    63→ * Sparc64
    64→ * Xtensa (ESP8266, LX106)
    65→
    66→Supporting new architectures requires:
    67→
    68→ 1. Add libc/machine/_architecture_ for architecture-specific libc
    69→    bits. This should at least include setjmp/longjmp support as these
    70→    cannot be performed in architecture independent code and are
    71→    needed by libstdc++. If available, it should also include thread
    72→    local storage setup code as libc/machine/_architecture_/tls.c; the
    73→    build system looks for that file by name when determining if TLS
    74→    is available.
    75→
    76→ 2. Checking for atomic support for stdio. Picolibc requires
    77→    atomics for ungetc to work correctly in a reentrant
    78→    environment. By default, it stores them in 16-bit values, but
    79→    some architectures only have 32-bit atomics. To avoid ABI
    80→    issues, the size selected isn't detected automatically, instead
    81→    it must be configured in libc/include/stdio.h.
    82→
    83→ 3. If necessary, add libm/machine/_architecture_ for any
    84→    architecture-specific math bits
    85→
    86→ 4. picocrt/machine/_architecture_ source code and build bits
    87→    for startup code needed for the architecture. Useful in all
    88→    cases, but this is necessary to run tests under emulation if your
    89→    platform can do that.
    90→
    91→ 5. cross-_gcc-triple_.txt to configure the meson cross-compilation
    92→    mechanism to use the right tools
    93→
    94→ 6. do-_architecture_-configure to make testing the cross-compilation
    95→    setup easier.
    96→
    97→ 7. run-_architecture_ script to run tests under QEMU. Look at the ARM
    98→    and RISC-V examples to get a sense of what this needs to do and
    99→    how it gets invoked from the cross-_gcc-triple_.txt configuration
   100→    file.

   102→## Relation to newlib
   103→
   104→Picolibc includes a lot of code originally from newlib. However, it
   105→has been reformatted and restructured to make it easier to read and
   106→maintain going forward.
   107→
   108→Here's a list of some differences between picolibc and newlib:
   109→
   110→ 1. Designed for and tested on 32- and 64- bit embedded
   111→    systems. Newlib (and newlib-nano) are used on both embedded
   112→    systems and Cygwin, with the bulk of the development focused on
   113→    improving the Cygwin support. Because the Cygwin version needs to
   114→    be very careful about ABI stability, many parts of the newlib code
   115→    base are very difficult to change. Picolibc has no such
   116→    constraints, so much work has been done to improve the ABI and API
   117→    for use in embedded systems.
   118→
   119→ 2. Stdio. Picolibc's stdio implementation borrows some code from
   120→    AVR-Libc along with substantial new code. The goal is to remain
   121→    compatible with relevant C and POSIX standards while using a very
   122→    narrow API to the underlying system (it only requires getc and
   123→    putc) along with doing no internal allocation and using a minimal
   124→    amount of RAM.
   125→
   126→ 3. Thread-local-storage support. Instead of creating a large data
   127→    structure containing all possible thread-specific data, Picolibc
   128→    uses the underlying TLS support from the compiler which only
   129→    allocates the amount of per-thread storage needed to support the
   130→    API called by the application. Typically, this means you use only
   131→    4 bytes for errno.
   132→
   133→ 4. Integrated test suite. Along with dozens of new tests, picolibc
   134→    includes tests from newlib and musl. These tests are run as a part
   135→    of the patch review process and new code or bugfixes are expected
   136→    to include tests that verify functionality and keep bugs from
   137→    recurring.
   138→
   139→ 5. Bare-metal startup code and linker scripts for many
   140→    architectures. These may not support precisely what your
   141→    application requires, but these provide a good basis for writing
   142→    your own startup code and linker scripts. These are used to create
   143→    bare-metal versions of the test suite which can then be run using
   144→    QEMU.
   145→
   146→ 6. Meson-based build system. This is largely used to avoid conflicts
   147→    with the newlib autotools build system files, but a side effect is
   148→    that the library builds very quickly.
   149→
   150→ 7. Narrow and well-defined POSIX OS requirements. Portions of
   151→    Picolibc which require underlying OS support are documented so
   152→    that you will know which functions are necessary based on the API
   153→    in use. In particular, picolibc requires no POSIX functions to
   154→    support stdin/stdout/stderr, and instead requires only functions
   155→    to get and put single characters to the target device.
   156→
   157→ 8. Built-in semihosting support for many architectures. This allows
   158→    you to take advantage of existing gdb, openocd and qemu
   159→    semihosting support for things like debugging output even before
   160→    you have a serial port working.
   161→
   162→ 9. Clear BSD licensing. All non-BSD compatible library source code
   163→    has been removed so that the status of the resulting library is
   164→    not in question.
