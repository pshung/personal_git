23
Instruction Throughput and Latency
This chapter lists instruction throughput and latency. The instruction throughput is the number of
cycles before executing the next independent instruction of the same kind. The instruction latency is
the number of cycles before executing the next instruction with read-after-write dependency.
Note
• For instruction latency, dependent instructions are assumed to be ALU instructions.
23.1
ALU Instructions
The latency and the throughput of ALU instructions are both 1 cycle. ALU instructions include:
• Add/Sub: ADD, SUB, ADDI, ADDW, SUBW, ADDIW
• Shift: SLL, SRL, SRA, SLLI, SRLI, SRAI, SLLW, SRLW, SRAW, SLLIW, SRLIW, SRAIW
• Logical: AND, OR, XOR, ANDI, ORI, XORI
• Compare: SLT, SLTU, SLTI, SLTIU
• LUI and AUIPC
• Load effective address instructions
• ADDIGP
• String processing: FFB, FFZMISM, FFMISM, FLMISM
• Bit ﬁeld operation: BFOS, BFOZ
• Bit-manipulation ISA extensions
– Zba and Zbs instructions
– Logical with negate: ANDN, ORN, XNOR
– Integer minimum/maximum: MAX, MAXU, MIN, MINU
– Sign-extension and zero-extension: SEXT.B, SEXT.H, ZEXT.H
– Bitwise rotation: ROL, ROLW, ROR, RORI, RORIW, RORW
– OR combine: ORC.B
– Byte-reverse: REV8

23.15
Vector Instructions
The instruction throughput and latency of a vector instruction is dependent on the settings of LMUL
and SEW ﬁelds in the vtype CSR. The term LMUL and SEW in the section speciﬁcally refer to the
vtype CSR ﬁelds, instead of EMUL and EEW of the source and destination operands. The throughput
of a vector instruction means the number of cycles needed for the execution of a vector instruction.
The result latency of a vector instruction signiﬁes the number of cycles before the ﬁrst result of the
vector instruction becomes available for its subsequent instruction.
23.15.1
Dual-Issue Capability of Vector Instructions
Vector instructions are executed in function units:
• VALU: See Section 23.15.4
• VMAC: See Section 23.15.5
• VFMIS: See Section 23.15.6
• VLSU: See Section 23.15.7
• VMASK: See Section 23.15.8
• VPERMUT: See Section 23.15.9
• VDIV: See Section 23.15.10
• VFDIV: See Section 23.15.11
• VMAC2: See Section 23.15.5
• VACE: Andes Custom Extension - Vector instructions
• VSP: Andes Custom Extension - Streaming port instructions
AX45MPV can issue a vector instruction with another instruction as an instruction pair unless it en-
counters an issue restriction. The vector instruction can be either the ﬁrst or the second instruction of
the instruction pair. The instruction pair may consist of two vector instructions. The issue restrictions
include:
• The ﬁrst instruction is a Load/Store instruction and the second instruction is a vector instruction.
• The ﬁrst instruction is a Branch/Jump instruction and the second instruction is a vector instruction.
• The ﬁrst instruction is a VLSU instruction and the second instruction is a vector instruction.
• The ﬁrst instruction is a VSP instruction and the second instruction is a VLSU instruction.
• The ﬁrst instruction reads more than 2 integer registers.
• The ﬁrst instruction writes 2 integer registers.
• The ﬁrst instruction reads 3 ﬂoating-point registers and the second instruction reads 1 ﬂoating-
point register.
• The second instruction reads more than 2 integer registers.

23.15.7
VLSU Instructions
The throughput and latency of VLSU instructions are dependent on the instruction type, memory
type, and resource restrictions. In addition, the data width (VLSU_MEM_DW) for memory accesses
depends on the access region as listed below.
• When a memory access is within the HVM region, VLSU_MEM_DW equals DLEN.
• When a memory access is within a cacheable/non-cacheable region, VLSU_MEM_DW equals BIU_
DATA_WIDTH.
Cacheable vector load or store requests are sent to L2 without allocating the data into D-Cache. If
the requested data is already allocated in the D-Cache, AX45MPV will handle the data coherency as
follows:
• If a vector load hits the D-Cache, the data will be read from the D-Cache or L2.
• If a vector store hits the D-Cache, the data will be sent to L2, and the cache line is evicted from the
D-Cache.

23.15.7.2
VLSU Resource Restrictions
VLSU is equipped with the following resources to handle memory requests and vector load and store
data.
1. Conﬁgurable Number of VLSU Requests: This option determines the maximum number of mem-
ory requests.
2. Conﬁgurable Depth of VLSU Data Buffer:
• The size of each data buffer entry is VLEN. This data buffer serves as a temporary storage for
transferring data between VRF and memory.
• For vector load instructions, VLSU reserves a data buffer entry until the out-of-order re-
sponded data writes back to VRF in order.
• For vector store instructions, VLSU might release a data buffer entry after the store data is
sent out to the bus.
If any of the above resources is not available, VLSU will stop sending out the memory requests.
VLSU also has an 8-entry Vector Translation Lookaside Buffer (vTLB) to store recently used PTE,
PMA, and PMP. See Section 7.3.3 for details. Each entry in the vTLB covers a 4KiB address range,
and the replacement policy implemented is pseudo-LRU. If a miss occurs in the vTLB, it will introduce
an extra penalty, which varies depending on whether the virtual address translation is enabled or
disabled.
• When the translation is disabled, a penalty of 2 cycles is incurred in the VLSU pipeline.
• When the translation is enabled, the penalty is determined by the virtual address translation pro-
cess.
23.15.7.3
VLSU Memory Dependency
VLSU memory dependencies come in three ﬂavors:
• RAW dependencies: a read access and an in-ﬂight write transaction are in the same 64-byte region.
• WAW dependencies: a write access and an in-ﬂight write transaction are in the same 64-byte region.
• WAR dependencies: a write access and an in-ﬂight read transaction are in the same 64-byte region.
To resolve memory dependencies for both cacheable and non-cacheable memory, VLSU will stall any
read/write access after dependent in-ﬂight transactions until they are completed.
VLSU does not stall HVM requests, and the HVM should complete the memory requests in order. See
Section 3.6 for details.
